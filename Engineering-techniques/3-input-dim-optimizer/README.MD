# YOLO 输入分辨率对比：384×640 vs 640×640

在 YOLO 目标检测模型的训练与部署中，输入分辨率的选择对模型的性能、推理速度以及检测精度有着至关重要的影响。以下是关于 **$384 \times 640$（矩形分辨率）** 与 **$640 \times 640$（正方形分辨率）** 的详细对比分析。

## 1. 计算量与推理速度

这是两者最直观的区别。

*   **640 * 640**：
    *   总像素数为 409,600。
    *   作为标准的正方形输入，它是 YOLO 官方模型（如 YOLOv5s, YOLOv8n 等）最常用的默认尺寸。在相同的硬件环境下，推理计算量最大，速度相对较慢。
*   **384 * 640**：
    *   总像素数为 245,760。
    *   像素总量约为 640 * 640 的 **60%**。这意味着模型在卷积运算时的计算量减少了约 40%，推理速度会显著提升，显存占用也会相应降低。

**结论**：如果对实时性要求极高（如移动端部署、边缘计算设备），384 * 640 是更优的选择。

## 2. 图像几何特征：填充与畸变

由于大多数监控画面、视频流通常是 **16:9 的横向画面**（如 1920 * 1080），这两种分辨率在预处理阶段的处理方式截然不同。

*   **640 * 640（正方形）**：
    *   为了将矩形原图缩放填充进正方形，YOLO 默认会对图像进行 **Letterbox** 操作：即在图像上下方添加灰色或黑色的填充边框。
    *   **缺点**：引入了大量无效的背景信息，模型可能会学习到“黑边”特征，且计算资源被浪费在处理黑边上。
*   **384 * 640（矩形）**：
    *   该比例约为 3:5，更接近 16:9 或 4:3 的常规画面比例。
    *   在预处理时，通常只需要极少的填充甚至不需要填充即可直接缩放。
    *   **优点**：最大程度保留了原图的纵横比，减少了图像畸变，且有效信息密度更高，没有无意义的黑边计算。

# 2head 模型不同分辨率（384 vs 640）性能对比分析

本次测试对比了 `2head` 模型在不同输入分辨率下的性能表现。其中输入尺寸分别为 384x640 和 640x640。
*(注：像素总量从 245,760 增加到了 409,600，像素量增加了约 **1.67倍**)*

为了直观，以下分析均已将时间单位换算为**毫秒**。

## 1. 数据统计概览

## 2. 详细分析

### （1）绝对速度对比（均值）
* **推理阶段**：`384` 分辨率（24.22ms）比 `640` 分辨率（34.32ms）快了 **10.1ms**，推理速度提升了约 **29.4%**。
* **后处理阶段**：`384` 分辨率（2.25ms）比 `640` 分辨率（3.42ms）快了 **1.17ms**，速度提升了约 **34.2%**。
* **整体来看**：`384` 分辨率的端到端平均耗时仅为 `640` 分辨率的 **70%** 左右，绝对速度优势非常明显。

### （2）计算缩放效率（深度洞察）
* 分辨率从 384x640 变为 640x640，**计算量（像素数）增加了 1.67 倍**。
* 但实际上，推理时间仅增加了 1.41 倍（34.32 / 24.22），并未呈严格的线性增长。这说明模型的底层算子（大概率是 GPU 并行计算）利用率很高，非线性加速比表现不错。

### （3）稳定性对比（方差与标准差）
* **推理阶段**：出乎意料的是，`640` 分辨率的方差（0.24）反而比 `384` 分辨率（0.92）**更小、更稳定**。`384` 分辨率在第4次测试中出现了一次轻微的波动（26.07ms）。
* **后处理阶段**：`640` 分辨率的后处理极其稳定（方差仅 0.015），远好于 `384`（方差 0.038）。这可能是因为 640 分辨率产生的特征图/候选框数量更大，计算密集度更高，反而抹平了系统调度的微小抖动。
* **整体来看**：虽然 `384` 更快，但 `640` 在耗时表现上**更加坚如磐石**，几乎没有毛刺。

### （4）尾部延迟（P99 极限情况）
* `384` 的最大耗时为 **28.62ms**。
* `640` 的最大耗时为 **38.87ms**。
* 即使对比 `384` 的最坏情况和 `640` 的最好情况（37.23ms），`384` 依然具有约 8.6ms 的绝对领先优势。

---

## 3. 结论与建议

**结论：从纯性能（速度）角度来看，`2head 384` 模型完胜。**

**工程落地建议：**

1. **首选 384 分辨率**：如果你的业务场景（如目标检测、人脸识别、姿态估计等）在 384x640 的分辨率下，**精度指标（mAP、召回率等）能够满足底线要求**，那么**毫不犹豫地选择 384 分辨率**。它能为你节省近 30% 的算力，极大地提高并发吞吐量或降低硬件成本。
2. **何时选择 640 分辨率**：只有当业务需求强依赖高分辨率（例如：需要检测非常微小的目标、远距离的目标，或者 384 分辨率下精度掉点严重导致不可用）时，才应该切换回 `640` 分辨率。因为此时你是在**用 30% 的性能代价去换取精度的提升**。
3. **关于稳定性的释疑**：虽然 `640` 表现出更小的方差，但 `384` 的方差（1.03）在工程上依然属于“极其稳定”的范畴（波动不到 2ms），完全不需要为了追求这点方差优势去使用更慢的 `640` 模型。
