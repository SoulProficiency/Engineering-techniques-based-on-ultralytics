# 基于RK3588的YOLO检测头解耦与后处理优化实践
本仓库仅用于模型的导出，基于ultralytics 8.4.30
## 简介
在将YOLO系列模型部署至瑞芯微RK3588平台时，常面临NPU对复杂后处理算子支持不佳的问题。本方案针对原始YOLO模型（单输出融合结构，后处理置于NPU图内）在RK3588上效率低下的痛点，提出了一种**“检测头解耦 + CPU异构后处理”**的优化策略。通过重塑模型输出结构并合理分配算力，在保证精度的前提下，实现了推理速度的成倍提升。

---

## 优化方案对比

| 对比维度 | 原始方案 (NPU融合) | 优化方案 (解耦+CPU后处理) |
| :--- | :--- | :--- |
| **模型输出结构** | 1个融合输出张量 | 6个独立输出张量 (`reg1`, `cls1`, `reg2`, `cls2`, `reg3`, `cls3`) |
| **后处理执行位置** | NPU (内置在推理图中) | ARM CPU (C++/Python实现) |
| **单张图片耗时** | **~0.080 s** (约 12.5 FPS) | **~0.037 s** (约 27.0 FPS) |
| **性能提升幅度** | 基准 | **提速约 116% (耗时减少 53.7%)** |

## 技术原理解析

### 1. 痛点分析：为何原生NPU后处理慢？
原始YOLO结构通常将分类和回归结果拼接为单一的大张量输出，并将NMS（非极大值抑制）等后处理算子（yolo26虽然舍弃了NMS，但是其官方的后处理依旧效率较低）融合在NPU计算图中。RK3588的NPU虽然矩阵运算能力极强，但对于**包含大量动态分支、循环索引和张量掩码操作**的后处理算子（如NMS），其执行效率极低，甚至会导致NPU计算单元闲置等待，造成严重的性能瓶颈。

### 2. 优化核心：检测头切割解耦
通过修改模型导出逻辑（如修改ONNX图），在模型末端将三个尺度的检测头进行切割解耦，分离出独立的回归分支（`reg1/2/3`）和分类分支（`cls1/2/3`）。这种解耦结构去除了不必要的张量拼接与reshape操作，大幅减轻了NPU的无效计算负担。

### 3. 异构调度：NPU算特征，CPU做筛选
将解耦后的6个分支直接作为模型的输出，将后处理逻辑（置信度过滤、坐标解码、NMS）从NPU剥离，交由RK3588强大的4xA76核心（CPU）来执行。CPU在处理复杂的逻辑判断和动态循环时，由于无需经过NPU指令转换，实际执行效率远高于在NPU上“硬跑”后处理算子，从而实现了异构计算资源的最优分配。

## 结论
本优化方案通过“化整为零”的解耦思想与“扬长避短”的异构调度，成功将YOLO模型在RK3588上的单帧处理耗时从 **80ms** 压缩至 **37ms**，轻松跨越了实时性门槛（>25FPS）。该方案不仅适用于yolo26，亦可作为各类Anchor-free目标检测模型在端侧NPU部署时的通用性能优化范式。
